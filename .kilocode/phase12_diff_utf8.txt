-# BasterBoer ΓÇö Agent Instructions
-
-## Project Overview
-BasterBoer is a South African land management simulation in Godot 4.6 with C# (.NET) for simulation and GDScript for UI/events.
-
-## Coding Rules
-
-### C# (Simulation Layer)
-- **Data-Oriented Design**: Use `readonly struct` for value types. Pool objects. Zero per-frame allocations.
-- **Engine Independence**: Simulation code must NOT depend on Godot APIs. Use `GodotObject` only at the boundary.
-- **Thread Safety**: Godot APIs are single-threaded. Simulation can be threaded but must marshal back to main thread for Godot calls.
-
-### GDScript (UI Layer)
-- Keep GDScript thin ΓÇö delegate to C# systems.
-- Use typed GDScript where possible.
-- Signal-based communication between nodes.
-
-### Conventions
-- South African terminology: bakkie, koppie, vlei, stoep, boma, etc.
-- Currency: ZAR (South African Rand), format as `R 1,234.56`
-- Land: hectares
-- Seasons: Southern hemisphere (Dec-Feb = summer)
-
-## Build
-```bash
-dotnet build "Claude Game.sln"
-```
-

-- `docs/ARCHITECTURE.md` ΓÇö Full technical architecture

-## 1. The Three Bottlenecks ΓÇö Know Which One You're Hitting
-
-Before writing any system, identify which hardware resource it strains.
-
-### CPU ΓÇö Killed by quantity of work per frame
-- Runs all game logic: AI, simulation, pathfinding, physics
-- C# is 3ΓÇô10├ù faster than GDScript on tight loops ΓÇö use it for all simulation
-- Enemy: 200 animals each running individual AI every frame
-- Solution: One herd brain, 200 bodies following its result
-
-### GPU ΓÇö Killed by draw call count, not polygon count
-- Draws things on screen ΓÇö fast at drawing the same thing many times
-- Enemy: 1,000 trees each with unique meshes = 1,000 draw calls
-- Solution: 1,000 trees sharing one mesh = 1 draw call (instancing)
-- Low poly reduces data per draw call but NOT draw call count
-
-### RAM ΓÇö Killed by loading too much at once
-- Holds all currently loaded meshes, textures, nodes, audio, data
-- Enemy: Loading a 10,000 hectare world simultaneously
-- Solution: Only the area near the player is loaded as scene nodes.
-  Everything else is lightweight C# data objects.
-

-### How the 300 draw call budget is used:
-- Terrain chunks (3├ù3 grid): ~9 draw calls
-- Tree species (MultiMesh per species per chunk): ~30ΓÇô50 draw calls
-- Grass (MultiMesh per chunk): ~9 draw calls
-- Animal herds (MultiMesh per species): ~10ΓÇô20 draw calls
-- Rocks/shrubs/ground cover: ~15 draw calls
-- Buildings and infrastructure: ~20ΓÇô40 draw calls
-- Fence system (poles, sticks, wire per chunk): ~15ΓÇô30 draw calls
-- Water/river shaders: 0 additional (shader params on terrain)
-- Fire (1ΓÇô2 particle systems): ~2 draw calls
-- UI: ~20ΓÇô30 draw calls
-- Roads: 0 additional (terrain texture blend)
-- Ambient wildlife (MultiMesh per species): ~10 draw calls
-- Headroom: ~60ΓÇô100 draw calls
-
-Any single naive system that spawns unique nodes per entity will consume
-this entire budget alone. Do not do it.
-

-## 3. Instancing Rules ΓÇö Non-Negotiable
-
-Any object that appears more than once uses `MultiMeshInstance3D`. No exceptions.
-
-```csharp
-// WRONG ΓÇö unique node per tree

-    var treeNode = treeScene.Instantiate<Node3D>(); // hundreds of draw calls

-// CORRECT ΓÇö one MultiMesh for all trees of this species

-    Mesh = acaciaMesh,

-### Instancing applies to:

-- All animals of same species in a visible herd
-- Grass blades (one MultiMesh per chunk, wind shader on GPU)
-- Rock formations

-- Shrubs and ground cover
-- Ambient wildlife of same species (40 queleas = 1 draw call)

-Use per-instance shader parameters ΓÇö NOT separate meshes.

-```csharp

-## 5. Asset Rules
-
-### Polygon targets
-| Asset Type | Triangle Budget |
-|-----------|----------------|
-| Animals | 400ΓÇô750 triangles |
-| Trees | 200ΓÇô800 triangles |
-| Rocks | 50ΓÇô200 triangles |
-| Buildings (individual component) | 100ΓÇô400 triangles |
-| Fence pole / dropper | 30ΓÇô80 triangles |
-
-Low poly is a technical requirement, not just an aesthetic. More polygons per
-draw call means less headroom for draw call count.
-
-### Textures
-- All flora and terrain textures combined into atlases ΓÇö one atlas per biome
-- Maximum atlas resolution: 2048├ù2048
-- No unique material per object ΓÇö each unique material = one more draw call
-- Animal coat variation via shader colour parameters, NOT separate textures
-
-### Audio
-- All ambient sounds: `AudioStreamPlayer3D` with distance attenuation
-- Maximum 12 simultaneous 3D audio sources
-- Animal calls triggered by EventSystem or herd state ΓÇö not always-on proximity nodes
-- Bakkie music: single `AudioStreamPlayer3D` on bakkie node, attenuates with distance
-

-## 6. What Low Poly Does NOT Fix
-
-Low poly reduces VRAM per mesh and GPU work per draw call. It does NOT:
-- Reduce draw call count ΓåÆ instancing does that
-- Reduce CPU load from simulation ΓåÆ C# tick architecture does that
-- Reduce RAM pressure from too many nodes ΓåÆ chunking does that
-
-Low poly is one layer. It requires all the other layers to function.

-
-## When to use this skill
-Read this file before designing or writing ANY new system, class, or feature in
-BasterBoer. This defines how the game world works at an architectural level.

-## 1. The Fundamental Law ΓÇö Simulate vs Render
-
-> Everything in the world is simulated at all times.
-> Only what is near the player is rendered.
-
-This is not a guideline. It is the load-bearing wall of the entire architecture.
-Every system you write must respect this split.
-
-| Layer | What it is | Technology | Runs |
-|-------|-----------|------------|------|
-| **Simulation** | C# data objects ΓÇö herds, flora, economy, fire, water | Pure C# classes and structs | Always, every tick |
-| **Render** | Scene nodes ΓÇö meshes, particles, audio | Godot nodes + MultiMeshInstance3D | Proximity only, freed on unload |
-
-A buffalo herd 3km from the player is **real** ΓÇö it grazes, breeds, ages, and
-can be poached. It has zero scene nodes. It is a C# object. When the player
-drives within 150m, the render layer reads the herd data and spawns meshes.
-When the player leaves, those meshes are freed. The C# data object continues
-running undisturbed.
-
-**The test:** Can your new class be instantiated without Godot running?
-If yes, it belongs in the simulation layer. If no, examine why.
-

-
